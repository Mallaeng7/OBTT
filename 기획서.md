# OBTT — Rust+ 연동 디스코드 봇 + 웹 대시보드 기획서 및 개발 가이드

이 문서는 Rust 게임 서버와 Rust+ 컴패니언 API를 연동하는 **디스코드 봇 + 웹 대시보드** 통합 플랫폼 **OBTT**를 개발하기 위한 상세 기획서이자, 다른 AI 에이전트에게 그대로 전달할 수 있는 개발 프롬프트입니다.

> **벤치마킹 레퍼런스**
> - [rustplusplus](https://github.com/alexemanuelol/rustplusplus) — 오픈소스 셀프호스팅 Rust+ 디스코드 봇 (Node.js, rustplus.js 기반)
> - [RustPlusBot](https://bot.rustplus.io/) — 유료 호스팅형 Rust+ 봇 (웹 대시보드, 스팀 로그인, 플러그인 지원)

---

## 1. 프로젝트 개요

- **목표**: 디스코드 봇과 **웹 대시보드**가 동일한 백엔드를 공유하며 상호 연동되는 Rust+ 통합 관리 플랫폼 개발. 디스코드와 웹 어느 쪽에서 제어해도 실시간으로 상태가 동기화되어야 합니다.
- **구성 요소**:
  1. **디스코드 봇**: 알림, 팀챗 릴레이, 인게임/디스코드 명령어
  2. **웹 대시보드**: 브라우저에서 여러 서버를 시각적으로 모니터링·제어 (Steam OAuth 인증 필요)
  3. **공유 백엔드**: Rust+ 세션 관리, 이벤트 폴링, 데이터 저장을 두 클라이언트가 공유
- **앱 구성**: 프론트엔드 + 백엔드 + 디스코드 봇을 **하나의 Next.js 앱(단일 프로세스)** 으로 통합. `npm start` 한 번으로 전부 기동.
- **배포 환경**: **VPS** (Linux, Docker 없이 Node.js 직접 실행 + systemd/pm2)
- **멀티 서버**: 한 인스턴스에서 **여러 개의 Rust 서버**를 동시에 연결·관리 (서버별 세션 병렬 유지).
- **포지셔닝**: rustplusplus처럼 **무료·셀프호스팅**으로 운영하되, RustPlusBot의 편의 기능(웹 대시보드, 계산기, 자판기 검색, 트래커 패널)을 선별 이식. **AI 어시스턴트 기능은 범위에서 제외.**
- **주요 기능 요약**:
  1. Rust+ 서버 페어링 및 스마트 기기(스위치/알람/저장 모니터) 관리 — 다중 서버
  2. 서버 이벤트(카고, 헬기, 오일릭, 잠긴 상자 등) 실시간 알림
  3. 디스코드 ↔ 인게임 팀 채팅 양방향 릴레이
  4. 인게임/디스코드 채팅 명령어(`!pop`, `!events`, `!craft` 등)
  5. 자동 갱신 트래커 패널 + 웹 대시보드 위젯(서버 정보, TC 유지비, 팀원 상태, 이벤트, 인터랙티브 맵)
  6. Battlemetrics 연동 플레이어 추적
  7. Steam OAuth 기반 웹 대시보드 인증·권한 관리

## 3. 기술 스택 (권장)

### 단일 앱 프레임워크
- **프레임워크**: **Next.js** (App Router, TypeScript 권장) — 프론트엔드·백엔드 API·디스코드 봇을 **하나의 앱, 하나의 프로세스**로 통합
- **실행 구조**: Next.js **커스텀 서버**(`server.ts`) — HTTP 요청은 Next 핸들러로 넘기고, 같은 프로세스에서 디스코드 봇·Rust+ 세션·WebSocket 서버를 함께 기동. `npm start` 하나로 전부 실행.
- **실시간 통신**: `Socket.IO` (커스텀 서버에 attach) — 서버 상태/이벤트를 대시보드로 실시간 푸시

### 봇 / Rust+ 연동
- **디스코드 라이브러리**: `discord.js` v14 이상
- **Rust+ 통신**: `@liamcottle/rustplus.js` (WebSocket 기반 Rust+ 컴패니언 API 클라이언트)
- **푸시 수신(페어링용)**: FCM 리스너 (`@liamcottle/push-receiver` 또는 rustplus.js 내장 기능)
- **정적 게임 데이터**: 아이템/제작/재활용/내구도 테이블을 JSON으로 내장 (Rust Labs 데이터 덤프 활용)
- **외부 API**: Battlemetrics REST API (플레이어 추적)

### 웹 대시보드 (Next.js 내장)
- **프론트엔드**: Next.js 페이지(React) — 대시보드 UI, 지도 렌더링은 `Leaflet`/Canvas
- **백엔드 API**: Next.js Route Handler(`app/api/**`) — REST API, 인증 미들웨어 공용 적용
- **인증**: **Steam OAuth** (Steam OpenID 2.0 — `openid` 라이브러리로 직접 구현 또는 `next-auth` Steam 프로바이더) + `iron-session` 기반 암호화 쿠키 세션 (별도 세션 서버 불필요)

### 저장 및 배포
- **데이터베이스**: **SQLite3** (`better-sqlite3`) — 단일 파일 DB(`data/obtt.db`), WAL 모드. 외부 DB 서버 불필요.
- **환경 변수 관리**: `dotenv` (Next.js 내장 `.env` 로딩 활용)
- **배포**: **VPS(Linux), Docker 미사용** — Node.js 직접 실행. `npm run build` 후 `npm start`로 단일 프로세스 기동, systemd 또는 pm2로 상시 실행 관리, 앞단에 Caddy/Nginx로 HTTPS 리버스 프록시

## 4. 시스템 아키텍처

```
                          ┌──────── OBTT 단일 Next.js 앱 (VPS, 프로세스 1개) ────────┐
[Rust 서버 A] ←WS(Rust+)→ │  RustPlusManager (서버별 세션 병렬)                       │
[Rust 서버 B] ←WS(Rust+)→ │      │              │                                    │
[Rust 서버 N] ←WS(Rust+)→ │  PollingLoop    StateStore(이벤트버스) ──┬── DiscordBridge ──→ [디스코드]
                          │      │              │                   └── Socket.IO ─────→ [브라우저]
[Rust+ 앱] →FCM 푸시→      │  FcmListener    SQLite3(data/obtt.db)                     │
                          │                                                           │
                          │  Next.js: 페이지(대시보드 UI) + Route Handler(REST API)     │
                          │           + Steam OAuth 인증                               │
                          └───────────────────────────────────────────────────────────┘
```

- **RustPlusManager**: 연결된 **여러 Rust 서버**의 WebSocket 세션을 서버별로 병렬 유지·재연결 관리. 각 세션은 독립된 상태 컨텍스트를 가짐.
- **FcmListener**: 유저가 인게임에서 서버/기기 페어링 버튼을 누르면 FCM 푸시를 수신하여 해당 유저·서버에 등록.
- **PollingLoop**: Rust+ API는 이벤트 푸시가 제한적이므로, 맵 마커·팀 정보·시간 등을 주기적(기본 10초)으로 폴링하여 변화를 감지(이벤트 발생 판정).
- **StateStore / 이벤트 버스**: 모든 서버 상태·이벤트를 중앙에서 관리하며, 변경 시 `DiscordBridge`와 `Socket.IO` **양쪽에 동일하게 브로드캐스트**하여 디스코드와 웹의 상태를 실시간 동기화. (예: 웹에서 스위치를 켜면 디스코드 제어 Embed도 즉시 갱신, 반대도 동일)
- **DiscordBridge**: 슬래시 명령어, 버튼/모달 상호작용, 트래커 패널 갱신, 팀챗 릴레이 담당.
- **Next.js 레이어**: 대시보드 페이지(React)와 REST API(Route Handler)를 같은 앱에서 제공. API는 Steam OAuth 세션·권한 검사 후 StateStore/RustPlusManager를 직접 호출(별도 API 서버·프록시 없음).
- **커스텀 서버(`server.ts`)**: 프로세스 진입점. Next.js 핸들러 + Socket.IO + 디스코드 봇 + RustPlusManager + FcmListener를 순서대로 초기화. `npm start` = 이 서버 하나 실행.

## 5. 환경 변수 (`.env`) 설정

- **필수 (봇)**
  - `DISCORD_TOKEN`: 디스코드 봇 토큰
  - `DISCORD_CLIENT_ID`: 디스코드 애플리케이션 클라이언트 ID
- **필수 (웹 대시보드 / 인증)**
  - `WEB_BASE_URL`: 대시보드 공개 URL (예: `https://obtt.example.com`) — Steam OAuth 콜백 구성에 사용
  - `PORT`: 앱 포트 (기본값: `3000` — 웹·API·Socket.IO 모두 이 포트 하나 사용)
  - `STEAM_API_KEY`: Steam Web API 키 (OAuth 프로필 조회용, [steamcommunity.com/dev/apikey](https://steamcommunity.com/dev/apikey))
  - `STEAM_RETURN_URL`: Steam OpenID 콜백 URL (예: `https://obtt.example.com/auth/steam/return`)
  - `SESSION_SECRET`: 세션/JWT 서명 시크릿 (충분히 긴 랜덤 문자열)
  - `ADMIN_STEAM_IDS`: 최고 관리자 SteamID64 목록 (쉼표 구분)
- **선택**
  - `LANGUAGE`: 봇 출력 언어 (기본값: `ko`)
  - `POLLING_INTERVAL_MS`: Rust+ 폴링 주기 (기본값: `10000`)
  - `BATTLEMETRICS_TOKEN`: Battlemetrics API 토큰 (Phase 2)
  - `DATABASE_PATH`: SQLite DB 파일 경로 (기본값: `data/obtt.db`)

> Rust+ 접속 크리덴셜(스팀 계정 연동 FCM 크리덴셜)은 `.env`가 아니라 유저별 등록 절차(`/credentials` 명령어 또는 웹 대시보드)로 수집하여 SQLite `credentials` 테이블에 저장합니다.

## 6. 데이터 저장 구조 (SQLite3)

- **DB**: 단일 파일 `data/obtt.db` (`better-sqlite3`, WAL 모드 활성화). 앱 최초 기동 시 마이그레이션 스크립트로 테이블 자동 생성.
- **정적 게임 데이터**(아이템/제작/재활용/내구도)는 DB가 아닌 `static/` 디렉터리의 JSON으로 내장(읽기 전용).

### 테이블 스키마

| 테이블 | 주요 컬럼 | 설명 |
|---|---|---|
| `users` | `steam_id`(PK), `persona_name`, `avatar`, `role`(admin/manager/viewer), `linked_discord_id`, `first_login_at`, `last_login_at` | 웹 대시보드 계정 (Steam OAuth 프로필 + 권한) |
| `user_guilds` | `steam_id`(FK), `guild_id` | 유저가 접근 가능한 길드 매핑 (N:M) |
| `guilds` | `guild_id`(PK), `language`, `events_channel_id`, `alarms_channel_id`, `teamchat_channel_id`, `trackers_channel_id` | 디스코드 길드 설정 |
| `rust_servers` | `id`(PK), `guild_id`(FK), `title`, `ip`, `port`, `player_id`, `player_token`, `is_active` | 등록된 Rust 서버 — **한 길드에 여러 행 = 다중 서버** |
| `devices` | `id`(PK), `server_id`(FK), `entity_id`, `type`(switch/alarm/storage_monitor), `name`, `group_name`, `state` | 스마트 기기 (스위치/알람/저장 모니터 통합) |
| `credentials` | `discord_user_id`(PK), `fcm_credentials`(JSON), `updated_at` | 유저별 FCM/Rust+ 크리덴셜 |
| `tracked_players` | `id`(PK), `server_id`(FK), `name`, `battlemetrics_id` | Battlemetrics 추적 대상 (Phase 2) |
| `vending_watch` | `id`(PK), `server_id`(FK), `item_name` | 자판기 아이템 감시 목록 (Phase 2) |
| `event_log` | `id`(PK), `server_id`(FK), `event_type`, `occurred_at` | 이벤트 발생 기록 — `!events` 경과 시간 계산용 |

- `rust_servers`가 길드당 여러 행을 가지므로 **여러 Rust 서버를 동시에 등록·연결**할 수 있으며, 각 서버는 독립된 세션·기기 목록을 가집니다. 디스코드에서는 `is_active` 서버가 명령어 대상이 되고, 웹 대시보드에서는 모든 서버를 사이드바로 동시에 조회할 수 있습니다.
- `better-sqlite3`는 동기 API라 단일 프로세스에서 별도 락 처리 없이 안전하며, 쓰기는 트랜잭션(`db.transaction`)으로 묶어 일관성을 보장합니다.
- 백업은 DB 파일 1개 + `static/`만 복사하면 완료됩니다.

## 7. 상세 기능 요구사항

### Phase 1 — MVP (필수)

#### 7.1. 페어링 및 다중 서버 연결
- `/credentials set` — 유저가 FCM 크리덴셜을 등록하는 절차 안내(모달 또는 파일 첨부, 웹 대시보드에서도 가능).
- 인게임 Rust+ 메뉴에서 "서버 페어링" 시 FCM 푸시를 수신하여 서버 목록에 자동 등록, 디스코드에 등록 완료 Embed 전송.
- **여러 Rust 서버를 동시에 등록·연결** 가능. 각 서버는 독립 WebSocket 세션으로 병렬 유지.
- `/server list`, `/server connect <서버>`, `/server disconnect`, `/server active <서버>` — 다중 서버 관리 및 명령어 대상 서버 전환.
- WebSocket 끊김 시 서버별로 지수 백오프(최대 5분) 자동 재연결하고 상태 변화를 로그 채널·대시보드에 알림.

#### 7.2. 스마트 기기
- 인게임에서 스마트 스위치/알람/저장 모니터 페어링 시 FCM 푸시로 자동 등록.
- **스마트 스위치**: 디스코드에 기기별 제어 Embed(이름, 상태, `켜기`/`끄기`/`토글` 버튼) 자동 생성. `/switch group` 명령으로 그룹 묶음 제어 지원.
- **스마트 알람**: 알람 트리거 시 지정 채널에 @멘션 포함 경고 Embed + 인게임 팀챗 알림.
- **저장 모니터(TC)**: TC 유지비 잔여 시간 조회, 임계치(예: 24시간) 미만 시 경고 알림.

#### 7.3. 서버 이벤트 알림
맵 마커 폴링으로 아래 이벤트를 감지하여 이벤트 채널과 인게임 팀챗에 알림:
- 카고쉽 출현/정박/퇴장, 순찰 헬기 출현/격추, CH47(친누크) 출현, 잠긴 상자 드롭, 소형/대형 오일릭 상자 해금 시작, 이동 상인(Travelling Vendor).
- 각 이벤트의 "마지막 발생 후 경과 시간"을 기록하여 `!events` 명령으로 조회 가능하게 합니다.

#### 7.4. 팀 채팅 릴레이
- 인게임 팀챗 메시지를 지정 디스코드 채널로 전달(발신자 이름 포함).
- 해당 디스코드 채널에 올라온 메시지를 인게임 팀챗으로 전송.
- 릴레이 채널에서 인게임 명령어(`!`)도 동작해야 합니다.

#### 7.5. 인게임/디스코드 공용 명령어 (접두사 `!`)
| 명령어 | 기능 |

## 8. 웹 대시보드 (핵심)

디스코드 봇과 **동일한 공유 백엔드**를 사용하며, 한쪽에서의 제어가 다른 쪽에 실시간 반영됩니다. VPS에 배포되고 HTTPS로 서빙됩니다.

### 8.1. 인증 — Steam OAuth
- 로그인은 **Steam OpenID(OAuth) 로그인 버튼** 하나로 시작. Next.js Route Handler에서 Steam OpenID 인증 후 SteamID64·닉네임·아바타를 수신하여 `users` 테이블에 저장(upsert).
- 인증되지 않은 요청은 모든 대시보드 페이지·API에서 차단 — Next.js `middleware.ts`에서 세션 검사 후 로그인 페이지로 리다이렉트(API는 401).
- 세션은 `iron-session` 암호화 쿠키로 유지(`SESSION_SECRET`으로 암호화, 별도 세션 저장소 불필요). HTTPS 전용(`Secure`, `HttpOnly`, `SameSite` 설정).
- **권한(Role)**:
  - `admin` — 모든 길드·서버 관리, 유저 권한 부여 (`ADMIN_STEAM_IDS`에 포함된 계정이 초기 admin)
  - `manager` — 지정된 길드의 기기 제어·설정 변경
  - `viewer` — 읽기 전용 모니터링
- 스팀 계정과 디스코드 계정 연동(선택): 디스코드에서 발급한 일회용 코드를 대시보드에 입력해 `linkedDiscordId` 매핑.

### 8.2. 대시보드 화면 구성
- **서버 선택 사이드바**: 등록된 **여러 Rust 서버**를 목록으로 표시, 연결 상태(온라인/재연결 중) 표시.
- **서버 개요**: 인원/큐, 인게임 시간, 맵 정보, 와이프 경과일 — WebSocket으로 실시간 갱신.
- **스마트 기기 패널**: 스위치 on/off/토글, 그룹 제어, 알람 상태, 저장 모니터(TC) 유지비 — 버튼 클릭 시 백엔드가 Rust+로 명령 전송.
- **이벤트 타임라인**: 카고·헬기·오일릭 등 이벤트를 실시간 스트림으로 표시.
- **팀원 현황**: 온라인 여부·좌표·AFK 시간.
- **계산기/자판기 검색**(Phase 2): 인게임 명령어와 동일 데이터를 웹 UI로 제공.
- **인터랙티브 맵**(Phase 3): 실시간 마커·히트맵.

### 8.3. 실시간 동기화
- 백엔드 `StateStore`가 상태 변경 시 `Socket.IO`(브라우저)와 `DiscordBridge`(디스코드)에 동시 브로드캐스트.
- 예: 웹에서 스위치를 켜면 → Rust+ 명령 전송 → 상태 변경 → 디스코드 제어 Embed와 다른 웹 세션이 즉시 갱신.

### 8.4. 웹 API 개요 (REST)
| 메서드·경로 | 인증 | 기능 |
|---|---|---|
| `GET /auth/steam` · `/auth/steam/return` | 공개 | Steam OAuth 로그인/콜백 |
| `POST /auth/logout` | 세션 | 로그아웃 |
| `GET /api/me` | 세션 | 내 프로필·권한 |
| `GET /api/servers` | viewer+ | 등록된 서버 목록·상태 |
| `GET /api/servers/:id/state` | viewer+ | 서버 상세 상태(인원·시간·이벤트) |
| `POST /api/servers/:id/switches/:eid` | manager+ | 스위치 제어(on/off/toggle) |
| `GET /api/servers/:id/team` | viewer+ | 팀원 현황 |
| `POST /api/credentials` | 세션 | 본인 FCM 크리덴셜 등록 |
| `GET/POST /api/admin/users` | admin | 유저 권한 조회·부여 |

- 모든 `/api/*`는 세션 미들웨어로 인증 검사, 권한이 부족하면 403.
- 상태 조회·제어 이벤트는 WebSocket 채널(`server:<id>`)로도 구독 가능.

## 10. 에러 핸들링 및 안정성

- Rust+ WebSocket 오류·타임아웃 시 유저에게 명확한 에러 메시지 출력(예: "서버가 오프라인이거나 Rust+ 연결이 끊어졌습니다"), 백그라운드에서 자동 재연결.
- FCM 크리덴셜 만료 감지 시 해당 유저에게 DM으로 재등록 안내.
- 모든 이벤트 콜백·상호작용 핸들러는 `try-catch`로 감싸 봇 프로세스가 죽지 않도록 하고, 오류는 파일 로그 + 관리 채널에 기록.
- SQLite는 WAL 모드로 열고, 다중 행 변경은 트랜잭션(`db.transaction`)으로 묶어 일관성 보장. 프로세스 종료 시그널(SIGINT/SIGTERM)에서 봇·세션 정리 후 DB를 정상 close하는 graceful shutdown 처리.
- Discord rate limit 대응: 트래커 패널 갱신은 변화가 있을 때만 edit, 알림은 채널별 큐로 순차 전송.
- 크리덴셜 값은 절대 로그에 출력하지 않으며, `.gitignore`에 `data/`(DB 포함)와 `.env`를 포함.

## 11. VPS 배포 구성 (Docker 미사용)

- **대상 환경**: Linux VPS (Ubuntu 22.04+ 권장), Node.js 20 LTS 이상 직접 설치.
- **프로세스 구성**: 앱은 **단일 Node.js 프로세스 1개**뿐입니다 (Next.js 커스텀 서버 = 프론트 + API + Socket.IO + 디스코드 봇 + Rust+ 세션).

| 구성 요소 | 역할 |
|---|---|
| OBTT 앱 (`npm start`) | 단일 프로세스 — 웹·API·봇 전부 |
| systemd 서비스 또는 pm2 | 상시 실행, 크래시 시 자동 재시작, 부팅 시 자동 기동 |
| Caddy 또는 Nginx | HTTPS 종단(Let's Encrypt 자동 갱신), `:3000`으로 리버스 프록시 (WebSocket 업그레이드 포함) |

- **배포 절차**: `git clone` → `.env` 작성 → `npm install` → `npm run build` → `npm start` (systemd/pm2 등록). 업데이트는 `git pull && npm install && npm run build` 후 재시작 한 번.
- **네트워크/보안**:
  - 외부에 노출하는 포트는 `80/443`(리버스 프록시)만. 앱 포트(`3000`)는 localhost 바인딩.
  - Steam OAuth 콜백(`STEAM_RETURN_URL`)이 동작하려면 도메인 + HTTPS가 필수이므로, 배포 시 도메인 연결과 인증서 발급을 초기 설정에 포함.
  - `data/obtt.db`와 `.env`는 일 1회 크론 백업 권장 (DB는 파일 1개 복사로 끝).

## 12. 개발 지시 사항 (프롬프트 마무리)

위의 기획 내용과 아키텍처를 바탕으로 **Phase 1(MVP: 디스코드 봇 + 웹 대시보드 기본 + Steam OAuth 인증)** 을 처음부터 완전히 동작 가능한 상태로 작성해주세요.

- **Next.js 기반 단일 앱**(프론트 + 백엔드 API + 디스코드 봇, 프로세스 1개)으로 구성하고, **`npm start` 한 번으로 전부 기동**되어야 합니다. `package.json` 스크립트는 `dev`(개발), `build`, `start`(커스텀 서버 실행), `register-commands`(슬래시 명령어 등록)를 포함해주세요. Docker는 사용하지 않습니다.
- 데이터베이스는 **SQLite3**(`better-sqlite3`, 단일 파일 `data/obtt.db`)를 사용하고, 앱 최초 기동 시 테이블이 자동 생성되도록 마이그레이션을 포함해주세요.
- 명령어 핸들러, 이벤트 핸들러, RustPlusManager(다중 서버 세션), FcmListener, StateStore(이벤트 버스), Socket.IO 게이트웨이, DB 저장소(Repository) 유틸리티를 모듈로 분리한 클린 아키텍처로 구성해주세요.
- 디스코드와 웹 대시보드 어느 쪽에서 제어해도 **양쪽 상태가 실시간 동기화**되는 것을 최우선 요구사항으로 구현해주세요.
- 웹 대시보드의 모든 페이지·API는 Steam OAuth 인증 없이는 접근할 수 없어야 하며, 권한(admin/manager/viewer) 검사를 미들웨어로 일관 적용해주세요.
- 여러 Rust 서버를 동시에 연결·관리할 수 있어야 하며, 서버 추가/제거가 런타임에 가능해야 합니다.
- AI 어시스턴트 관련 기능은 구현하지 마세요 (범위 제외).
- Phase 2·3 기능은 신규 모듈 추가만으로 확장할 수 있도록 명령어/폴링 훅 구조를 설계해주세요.
- 모든 유저 대상 출력 문자열은 언어 파일(`locales/ko.json` 등)로 분리하여 다국어 확장이 가능하게 해주세요.
